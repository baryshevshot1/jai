# Журнал изменений — Jarvis AI (`jai`)

Все значимые изменения проекта. Формат основан на [Keep a Changelog](https://keepachangelog.com/ru/);
проект придерживается принципов офлайн-ядра из [`CLAUDE.md`](./CLAUDE.md) (единственный

## [Не выпущено] — 2026-06-26

Крупный блок: работа с документами (прикрепление и полноценный RAG), операционный

### Added — Зрение и OCR (vision-модели)
Приложение понимает изображения: прикрепить картинку → спросить по ней или распознать
текст (OCR) — всё офлайн через vision-модель (`qwen2.5vl`, `qwen3-vl`).
- Rust: `ChatMessage.images: Vec<String>` (сырой base64, `skip_serializing_if` — в
  текстовых сообщениях не появляется); команда `read_image_base64` (без префикса
  `data:`, проверка типа/размера ≤12 МБ); `list_models` отдаёт capability `vision`.
  Зависимость `base64` (MIT/Apache). `chat_stream` не изменён — картинки едут в `messages`.
- Фронт: кнопка «Прикрепить изображение» + превью с «убрать»; base64 кладётся в
  `images` сообщения через `buildApiMessages`; изображение сохраняется в истории и
  рисуется в пузыре. Гейт vision-модели: текущая умеет → используется; не умеет, но
  есть другая → авто-переключение с уведомлением; нет вовсе → предложение установить.
- OCR: действие «Извлечь текст» — тот же vision-запрос с OCR-промптом на русском.
- Зрение **отделено от RAG** (картинки не эмбеддятся/не индексируются); сеть только из Rust.

### Added — Управление моделями: состояние, источник, проверка обновлений
Зрелое управление вместо бинарного «есть/нет»: видно состояние каждой нужной модели,
обновление по digest, выбор источника (онлайн или с диска).
- Rust: расширяемый список `MODEL_SET` (роль/тег/обязательность); команда
  `model_states` — локальное состояние (установлена + digest/размер/дата) без сети;
  `check_model_updates` — сравнение **sha256 манифеста из реестра** Ollama с локальным
  digest → актуальна / есть обновление / не установлена (онлайн, по запросу). Добавлен
  `rustls-tls` (чистый Rust, MIT/Apache) — единственный выход в интернет, по кнопке.
- Источники: **онлайн** (`pull_model` по тегу = установка ИЛИ обновление) и **с диска**
  (офлайн-поставка `set_models_dir`/`reload_engine`) — переиспользованы, не дублированы.
- UI: на странице настроек раздел «Модели» — список с состоянием (цветные бейджи),
  иконки ролей, действия по состоянию (Установить/Обновить), кнопка «Проверить
  обновления» с итогом прямо в кнопке (Актуально/зелёный, Есть обновления/жёлтый),
  вращением стрелок при проверке и сохранением итога на сессию. Прогресс/отмена при
  установке/обновлении; края (нет сети, внешний движок). Отступы между блоками настроек.

### Added — Документы: прикрепление файла (Фаза A) · `8ccb4f2`
- Команда Rust `extract_document(path) → { name, ext, text, chars }` — извлечение
  текста из **PDF / DOCX / TXT / MD** чистым Rust, без сети и без Python/C-движков
  (`pdf-extract`, `quick-xml`, `zip` c фичей `deflate`). Скан-PDF без текстового
  слоя → понятная ошибка (OCR — отдельный этап).
- Плагин `tauri-plugin-dialog` для нативного выбора файла; разрешение `dialog:default`.
- Прикреплённый файл визуализируется карточкой в пузыре сообщения (бейдж формата
  PDF/DOCX/TXT/MD + имя + размер); при отправке поле ввода очищается, файл
  «уезжает» вместе с вопросом. Документ привязан к сообщению (`Message.doc`),
  остаётся в контексте для follow-up и переживает перезагрузку диалога; в Ollama
  отдельным полем не уходит (`skip_serializing_if`).

### Added — Документы: база и семантический поиск (RAG, Фаза B) · `d487f15`
Документы превращаются в постоянную базу с поиском по смыслу; в модель уходят не
целые файлы, а найденные релевантные фрагменты. Всё офлайн.
- **B1 — Векторное хранилище** (`src-tauri/src/docstore.rs`): `rusqlite` (feature
  `bundled`) + `sqlite-vec`; БД `appDataDir/documents.db`; виртуальная таблица
  `vec0` с **косинусной** метрикой; таблицы `documents`/`chunks`/`meta`; версия
  схемы (`PRAGMA user_version`) с безопасным пересозданием; защита от смены
  размерности вектора. Юнит-тесты на регистрацию расширения и KNN.
- **B2 — Эмбеддинги** (`src-tauri/src/embed.rs`): `bge-m3` через Ollama `/api/embed`
  **батчем**; размерность определяется из ответа (1024), не хардкодится; понятное
  сообщение при отсутствии модели (HTTP 404). `keep_alive=30s` — модель не висит
  в памяти зря (см. «Оптимизация памяти»).
- **B3 — Чанкинг и индексация** (`src-tauri/src/chunk.rs`): резка по естественным
  границам (абзацы → предложения, перекрытие ~12 %, числа в константах);
  идемпотентность по **sha256** содержимого; прогресс индексации через `Channel`.
- **B4 — Поиск**: команда `search_documents(query, k) → RetrievedChunk[]` —
  эмбеддинг вопроса → KNN top-k с метаданными источника (документ, № фрагмента).
- **B5 — Встраивание в чат + интерфейс базы**: фрагменты вставляются как
  `CONTEXT` **перед** вопросом в существующий `chat_stream` (поток не изменён);
  бюджет `num_ctx` соблюдается (история урезается при активном поиске); под
  ответом — источники. Интерфейс — **вкладки сайдбара «Диалоги | Документы»**
  (список, добавление, удаление, прогресс), отдельно от истории диалогов.
- **B6 — Честная обработка краёв**: пустая база → обычный чат; нет `bge-m3` →
  подсказка/установка; вопрос не по документам → обычный ответ; битый файл →
  понятная ошибка; смена модели эмбеддингов → пересоздание базы с предупреждением.

### Added — Операционный слой: установка моделей · `8910c1d`
- Команда Rust `pull_model(name, on_event, cancel)` — установка модели через
  Ollama `/api/pull` (stream) с **прогрессом** (`Channel<PullEvent>`) и **отменой**
  (`cancel_pull`, отдельный `PullCancelFlag` — не конфликтует с отменой чата).
  Разбор потока — тем же байтовым способом, что `chat_stream` (NDJSON, UTF-8 на
  границе чанков, дочитка хвоста); `chat_stream` не изменён.
- Фронт: во вкладке «Документы», когда `bge-m3` отсутствует — карточка
  «Установить bge-m3 (~1.2 ГБ)» с индикатором (проценты/объём, статусы по-русски)
  и кнопкой «Отмена»; после успеха список моделей и статус базы обновляются без
  перезапуска. Края: Ollama недоступна / нет интернета / отмена — понятные
  сообщения (офлайн-поставка моделей с диска — отдельный поздний механизм).
- Формат `/api/pull` сверен с живой Ollama; карточка установки «вживую» не
  прогонялась, т.к. `bge-m3` уже установлена на машине разработки.

### Added — Операционный слой: управление процессом Ollama
Приложение само обеспечивает работу движка — пользователю не нужен терминал.
- Модуль Rust `engine.rs` + команда `ensure_engine`: при старте **переиспользует**
  запущенную Ollama либо **поднимает свою** (`ollama serve` с `OLLAMA_HOST=127.0.0.1:11434`),
  дожидаясь готовности (опрос `/api/version`, до ~30 с на холодный старт).
- **Останавливает только тот процесс, что запустило само** (`started_by_us`), на
  выходе (`RunEvent::Exit` → `stop_if_ours`); внешний/системный экземпляр не трогает.
  Гашение со всеми дочерними: Unix — группа процессов (`process_group` + `SIGTERM`,
  ожидание ~3 с, затем `SIGKILL`); Windows — `taskkill /T /F`.
- **Гибкое разрешение путей** к движку без захардкоженного абсолютного пути: override
  из настроек (`ollama_path`/`ollama_models_dir`) → ресурс рядом с приложением → PATH
  и типичные места (последние — как fallback под `#[cfg(target_os)]`). Задел под
  будущую офлайн-поставку (инсталлер заполнит override).
- Каталог моделей: при отсутствии override **явно убираем унаследованную** из
  окружения `OLLAMA_MODELS` — чтобы движок использовал свой каталог по умолчанию,
  как системный (иначе поднятый движок не видел моделей).
- UI: статус «Запуск движка…»; края `not_installed`/`error` — честно.
- Проверено вживую на macOS: переиспользование, запуск своего движка, гашение
  только своего процесса (без осиротевших), грациозная остановка.
- Зависимости: `tokio` (feature `time`), `libc` под `cfg(unix)` — MIT/Apache.

### Added — Офлайн-поставка движка и моделей с диска
Замыкает офлайн-контур: у air-gapped клиента приложение находит движок и модели на
диске и работает без интернета. Опирается на гибкое разрешение путей из `engine.rs`.
- Команды Rust: `set_engine_path` / `set_models_dir` (с валидацией — исполняемый
  файл по биту исполнения/`.exe`; каталог моделей по наличию `manifests`+`blobs`),
  `clear_engine_overrides`, `reload_engine` (перезапуск **нашего** движка с новым
  `OLLAMA_MODELS`; внешний/системный не трогаем — честно «применится при запуске»).
  Пишут в те же ключи, что читает `engine.rs`. Юнит-тесты валидаторов.
- Офлайн-установка моделей: указать предзагруженный каталог Ollama → запись override
  → движок поднимается с ним; список моделей и статус эмбеддера обновляются. Без сети.
- Онлайн-провижининг (`pull_model`) не тронут — офлайн добавлен как альтернатива.

### Added — Страница настроек + полировка интерфейса
- **Страница настроек** на весь экран (на месте ленты диалогов): открывается
  иконкой-шестерёнкой в верхней панели, закрывается стрелкой «назад». Раздел
  «Движок и модели» (каталог моделей, путь к движку, сброс) со своей строкой статуса.
- **Левая панель**: регулируемая ширина перетаскиванием ручки (200–420px) и
  сворачивание кнопкой в топбаре; ширина и состояние сохраняются в `settings.json`.
- **Блок «Железо»**: центрированная карточка — статус → характеристики (перенос
  только по «·») → **рекомендованная модель прямо в блоке** (с разделителем и иконкой),
  без всплывающей подсказки.
- **Адаптив**: при сужении окна топбар/панель/чат корректно сжимаются (бренд
  скрывается, селект и статус движка ужимаются), без горизонтальной прокрутки.
- Иконки приведены к единому качественному виду (исправлена обрезка шестерёнки).

### Added — Настройки: долговечное хранение · `43350f2`
- Единый источник истины — `settings.json` (`appDataDir`): тема, **выбранная
  модель** (`selected_model`), **режим Thinking** (`thinking_enabled`).
- Выбранная модель сохраняется и восстанавливается между запусками; если её больше
  нет в списке — откат на дефолт без падения и без пустого выбора.
- Thinking перенесён из `localStorage` в `settings.json` с **одноразовой миграцией**
  старого ключа `jai.think` (значение не теряется); возможность модели уважается.

### Changed — UX работы с документами · `8910c1d`
- «Источники: …» под ответом показываются для каждого документа **один раз за
  диалог** (под первым ответом, где он использован), дальше не повторяются — в т. ч.
  в сохранённых диалогах; новый документ-источник показывается один раз.
- Прикреплённый файл подаётся модели как **справочный материал**, без приказа
  «отвечай только по нему» — модель обращается к нему по делу и не пересказывает
  его в каждом ответе (поведение, привычное по другим ассистентам).

### Performance — Оптимизация памяти · `d487f15`
- `bge-m3` выгружается из памяти через ~30 с после использования (`keep_alive`),
  а не висит стандартные 5 минут рядом с чат-моделью. На машине разработки
  замерено: приложение ~68 МБ, модели — основная нагрузка; рисков «вырубить
  ноутбук» нет (на целевом ПК 64 ГБ RAM — тем более).

### Соблюдение архитектурных правил
- Сеть — только из Rust (`reqwest`, `default-features=false`), строго `127.0.0.1`;
  фронтенд в сеть не ходит.
- Хранение — `appDataDir` через Tauri path API (`documents.db`, `settings.json`,
  `conversations/`); запись настроек атомарна (temp+rename) и под мьютексом.
- `chat_stream` (стриминг чата) при всех доработках не изменён — RAG и установка
  моделей встроены вокруг него.
- Кроссплатформенно: C-зависимости (`bundled` SQLite, `sqlite-vec`) собираются в
  CI-матрице (Linux/Windows/macOS); локально — только macOS.
