Инженерный отчёт · офлайн ИИ-ассистент · Tauri 2 + Ollama

Финальные решения и набор моделей

Сводный технический отчёт по продукту: три ключевых решения (движок, русские модели, набор моделей), операционный слой, соответствие 152-ФЗ, лицензии и дорожная карта. Все выводы проверены по официальным источникам 2026 года и согласованы с итоговым руководством по разработке.

Продукт: боксовый офлайн-ассистент для российского бизнеса · Стек: Tauri 2 (Rust + TypeScript), единственный движок Ollama · Процесс: разработка на macOS, выпуск Linux + Windows через GitHub Actions · Ориентир железа: NVIDIA 6 ГБ VRAM / 64 ГБ RAM.

◆Резюме

Три решения определяют архитектуру и набор моделей продукта.

Решение 1 · Движок
Один Ollama в MVP
Второй движок (llama.cpp) не нужен: Ollama построен поверх него и крутит те же модели. llama.cpp оправдан только для речи — Стадия 2.
Решение 2 · Русский
Гибрид, не обязателен
База — Qwen (хорошо справляется с русским). Русский профиль T-lite подключается опционально для ролей с интенсивным деловым текстом.
Решение 3 · Набор
Зафиксирован
Базовый набор — Qwen3.5-9B + bge-m3 + sqlite-vec. Зрение/код/русский — под роль. Мощное железо и речь — отдельно.

Подход признан рабочим и эффективным для MVP. Главное, что было исправлено относительно ранних версий: канал поставки русских моделей (они на Hugging Face, а не в реестре Ollama) и операционный слой (хранение, обновление, диагностика, безопасность), которого раньше не хватало. Эти выводы уже внедрены в руководство по разработке.

◆Контекст и задача

Что за продукт и какие ограничения формируют решения.

Продукт — десктоп-приложение, которое после установки работает полностью офлайн: данные не покидают компьютер клиента (152-ФЗ), у клиента нет терминала. Бизнес-модель — установка под ключ. Разработка ведётся только на macOS через Claude Code; продукт выпускается под Linux и Windows, сборка установщиков — через GitHub Actions, поскольку Tauri не умеет кросс-компиляцию.

Эти ограничения задают рамку для всех технических решений: минимум точек отказа на чистой машине клиента, предсказуемая офлайн-работа, простота сопровождения не-программистом, и приоритет моделей с открытой лицензией, доступных из РФ.

1Решение 1: движок — один Ollama

Второй движок llama.cpp в MVP не добавляем. Разбор честный, без предвзятости.

Ключевой факт: Ollama построен поверх llama.cpp и запускает те же GGUF-модели. Поэтому «добавить llama.cpp» не означает «получить другие или лучшие модели» — это лишь второй движок ради тонкого контроля и отдельных фич.

АспектЧто даёт llama.cpp сверх OllamaНужно ли нам в MVP
МоделиНичего — те же GGUFНет
Скорость (один пользователь)Быстрее лишь на ~3–10% на NVIDIAНесущественно
Тонкий контроль инференса, грамматикиБольше флагов; structured output в Ollama закрывается параметром formatИзбыточно
Речь (STT/TTS)Ollama не умеет речь вообще; нужен whisper.cpp + Piper/KokoroТолько Стадия 2

Цена второго движка для боксового офлайн-продукта высока: второй процесс и порт, отдельный провижининг под три ОС, рост размера установщика и новый класс багов (особенно Windows с видеокартами AMD/Intel). Для не-программиста-заказчика и слабого железа это явный минус.

Вывод

В MVP — один движок Ollama с аккуратным провижинингом (обнаружить/установить, проверить версию, корректно гасить группу процессов). Второй движок (llama.cpp) подключаем только когда в объём войдёт речь — это Стадия 2.

2Решение 2: русские модели — гибрид

Отдельные русские модели не обязательны, но дают измеримую пользу в нишевых ролях.

Общие модели Qwen3.5/Qwen3.6 многоязычны и уверенно работают с русским деловым текстом. Специализированные русские модели T-tech дают прибавку прежде всего за счёт плотного кириллического токенизатора (меньше токенов на русский текст → быстрее инференс и больше полезного контекста) и за счёт tool-calling. По независимому русскому бенчмарку прибавка измеримая, но небольшая.

ВариантПлюсыМинусы

Вывод

Базовая текстовая модель — Qwen (реестр Ollama, Apache 2.0). T-lite-it-2.1 подключается опционально как «русский профиль» для ролей с интенсивным русским деловым текстом (бухгалтерия, юристы, документы). T-pro-it-2.1 — только для мощного железа. Так берём выгоду русской специализации без жёсткой зависимости всего продукта от Hugging Face.

Зафиксированный набор по категориям, с которым нет проблем по доступности, лицензии и железу. Ориентир «влезает в 6 ГБ VRAM» — квант Q4_K_M при умеренном контексте.

Стадия 2 позже

Речь: whisper.cpp (STT) + Piper/Kokoro (TTS) — там же второй движок.

Зрение и OCR — одной моделью

qwen3-vl закрывает и анализ изображений, и распознавание текста/таблиц (в т.ч. русский). Отдельный OCR-движок в MVP не нужен; топовый разбор очень сложных табличных сканов — Стадия 2 при недостаточном качестве.

◆Канал поставки моделей

Главная корректировка относительно ранних версий: два источника, разведённые явно.

Общие модели берутся из реестра Ollama — он доступен из РФ без VPN. Русские модели T-tech в реестре Ollama отсутствуют — они опубликованы на Hugging Face и ставятся через синтаксис hf.co/.... Команда вида ollama pull t-tech/... не работает.

Установка моделей
# общие — из реестра Ollama
ollama pull qwen3.5:9b
# русские — с Hugging Face (НЕ ollama pull t-tech/...)
ollama run hf.co/t-tech/T-lite-it-2.1-GGUF:Q4_K_M
# постоянная модель через Modelfile
FROM hf.co/t-tech/T-lite-it-2.1-GGUF:Q4_K_M
ollama create t-lite-ru -f Modelfile
# офлайн-резерв с диска (сверить sha256)
FROM ./T-lite-it-2.1-Q4_K_M.gguf
Доступ к Hugging Face из РФ

Доступ нестабилен (в т.ч. из-за Xet-бэкенда). Поскольку русские модели — ядро ценности для РФ-бизнеса, для них предусмотрен резервный сценарий: офлайн-диск с заранее скачанными .gguf и установкой через FROM ./file.gguf со сверкой sha256.

◆Операционный слой

То, что делает продукт пригодным к эксплуатации у клиента. Этого слоя не хватало в ранних версиях — теперь он добавлен.

ОбластьРешение
ХранениеМодели Ollama — предсказуемый каталог через OLLAMA_MODELS; данные приложения (история, индекс sqlite-vec, настройки) — через Tauri path API (appDataDir/appConfigDir/appLogDir). Модели и данные переживают обновление; на диск заложить ~2–4 ГБ под движок + размер моделей.
ОбновлениеTauri updater подписывает апдейты своим ключом (отдельно от подписи кода ОС), endpoint собственный, проверка неблокирующая (без сети приложение работает). Модели — через /api/tags (не качать повторно), /api/pull (докачка слоёв), /api/delete (чистка).
ДиагностикаЛоги через tauri-plugin-log в appLogDir; встроенный экран «Проверка системы» (версия движка, модели, место, порт, RAM/VRAM, тест генерации, сбор логов). Понятные сообщения вместо «зависания».
БезопасностьOllama только на 127.0.0.1:11434 (никогда 0.0.0.0), сужённый OLLAMA_ORIGINS, без облака и телеметрии; обработка занятого порта; гасить только свою группу процессов; держать движок пропатченным.
Минимальная версия движка

В провижининге проверяется /api/version. Эффективный минимум — наибольшее из требований: зрение qwen3-vl требует ≥0.12.7, патчи безопасности — ≥0.17.1; практический ориентир — стабильная ветка 0.30.x. Если версия ниже — экран «обновите движок», без молчаливого обновления.

◆Железо и скорость

Честные ожидания, чтобы продукт не казался «зависшим», и пороги для светофора.

Зелёный

8–9B при ≥6 ГБ VRAM или ≥16 ГБ RAM. На ровно 6 ГБ — впритык: контекст ≤8k.

Жёлтый

3–4B при 4 ГБ VRAM или 8 ГБ RAM. Часть на процессоре — медленнее.

Красный

<4 ГБ или модель не влезает — своп, зависание. Заблокировано.

Ориентир скорости для модели 8B в Q4_K_M: около 40 токенов/сек на видеокарте RTX 3060, быстрее на старших картах. На процессоре без GPU — кратно медленнее (единицы токенов/сек). Формула оценки памяти: (размер файла × 1.15) + KV-кэш + ~0.5 ГБ; резерв ОС max(3 ГБ, 20% RAM). При отсутствии интернета на первом запуске продукт честно сообщает «модели не загружены» и предлагает подключить сеть или вставить офлайн-диск.

◆Соответствие 152-ФЗ

Инженерная проверка локальной обработки персональных данных (не юридическая гарантия).

Продукт обеспечивает техническую локальность: все вычисления и хранение — на ПК клиента, после установки нет исходящего трафика с ПДн, требование локализации (хранение в БД на территории РФ) выполняется автоматически.

Инженерный чек-лист

✓

Нет исходящего трафика с ПДн — проверить сетевым монитором после установки, зафиксировать в протоколе.

✓

Локальное хранение — история, индекс, файлы только на ПК (appDataDir).

✓

Телеметрия выключена — нет облачных вызовов и аналитики с содержимым.

✓

Обновления без ПДн — updater передаёт только версию/таргет.

✓

Удаление истории — пользователь может стереть данные.

Граница ответственности

Продукт даёт техническую локальность. Зона клиента-оператора (организационные меры): согласия субъектов, политика обработки ПДн, уведомление РКН, модель угроз. Это инженерный чек-лист со ссылкой на первоисточник (ФЗ-152), а не юридическая гарантия.

◆Дорожная карта

Что входит в MVP и что включается на Стадии 2 — с явными порогами.

Агентный режим — граница с 152-ФЗ

Офлайн-режим остаётся базовым и полностью локальным. Агентный режим — отдельная опция: сама модель в сеть не ходит, она через tool-calling запрашивает контролируемые инструменты (наш Rust-код выполняет действие и возвращает результат). Правила: выключен по умолчанию; включается явным тумблером пользователя; в интерфейсе видно, что использован интернет; в 152-ФЗ-чек-листе — отдельный раздел про данные, уходящие наружу с согласия пользователя. Реализуется после готовности офлайн-ядра.

◆Открытые вопросы
