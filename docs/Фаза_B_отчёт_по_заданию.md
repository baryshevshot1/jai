## 3. Подтверждённые факты окружения (проверены вживую)

- **Ollama 0.30.8** (`GET /api/version`). ⇒ Современный эндпоинт **`/api/embed`** (поле `input`:
  строка/массив → батч; ответ `embeddings: [[…]]`). Старый `/api/embeddings` не нужен.
- **`bge-m3` изначально НЕ установлена** на машине разработки. Более того, сервер при наборе
  моделей без эмбеддера **не отдаёт эмбеддинги вообще** (`"This server does not support
  embeddings"`) — для любого эмбеддинга нужна именно модель-эмбеддер. ⇒ Живой тест индексации/
  поиска возможен только после `ollama pull bge-m3` (~1.2 ГБ). По согласованию модель ставится.
- **Размерность вектора** не хардкодится: определяется из длины первого эмбеддинга и
  фиксируется в таблице `meta` (`embed_dim`). У bge-m3 ожидается 1024, но проверяется фактически.
- **Метрика косинус** — синтаксис `vec0(embedding float[N] distance_metric=cosine)` подтверждён
  по C-исходнику `sqlite-vec` 0.1.9 (допустимые значения: `l2`, `l1`, `cosine`).
- **Паттерн регистрации** расширения — `sqlite3_auto_extension(transmute(sqlite3_vec_init))` —
  взят из теста крейта; функция импортируется из `sqlite_vec` (с `#[link(name="sqlite_vec0")]`).

## 4. Расхождения задания с реальным кодом (учтены)

1. **Сигнатура `extract_document`.** Задание предполагает `{ filename, text, char_count }`.
   Фактически команда возвращает `DocumentText { name, ext, text, chars }` (поле `ext` добавлено
   в Фазе A). Беру реальные имена; `ext` полезен для метаданных документа.
2. **`chat_stream` шире approximate-сигнатуры** — есть `think` и `cancel`. На встраивание RAG
   это не влияет: `CONTEXT` добавляется во фронтовый массив `messages`, внутренности не трогаю.
3. **Сборка `messages` уже не «в лоб».** После доработки Фазы A фронт собирает массив в
   `buildApiMessages()`. Вставку `CONTEXT` делаю там же — единая точка, без дублирования логики.
4. **Номера страниц.** Задание просит «по возможности стр./№ фрагмента». `pdf-extract`
   (используется в Фазе A) выдаёт сплошной текст без надёжных границ страниц. ⇒ Цитирование —
   по **имени документа + индексу фрагмента**; поле `page` в схеме есть, но пока `NULL`.
   Честно отмечается как ограничение, не выдаётся за точную пагинацию.
5. **Чанкинг: токены vs символы.** Задание даёт ориентир в токенах (~500–800). Токенизатора
   bge-m3 в Rust нет, тянуть его — лишняя зависимость. Использую символьный прокси
   (целевой размер ~2000 символов, перекрытие ~12%), параметры — в константы. Это безопасно
   укладывается в лимит входа bge-m3 (8192 токена) и настраивается позже под русский.

## 5. Технические решения

### Крейты и лицензии (всё разрешительное — соответствует CLAUDE.md)
| Крейт | Версия | Лицензия | Назначение |
|---|---|---|---|
| `rusqlite` (feature `bundled`) | 0.40.1 | MIT | SQLite, вкомпилирован (без системной зависимости) |
| `sqlite-vec` | 0.1.9 | Apache-2.0 / MIT | C-расширение `vec0` (KNN по векторам) |
| `zerocopy` | 0.8.52 | BSD-2 / Apache-2.0 / MIT | `Vec<f32>` → BLOB без копий |
| `sha2` | 0.11.0 | MIT / Apache-2.0 | sha256 для идемпотентности |

Лицензии задокументированы комментариями в `Cargo.toml` (как уже сделано для Фазы A).

### Схема БД (`appDataDir/documents.db`, отдельный файл от диалогов)
- `PRAGMA user_version = 1` — версия схемы; при несовпадении индекс пересоздаётся (это кэш
  поверх исходных файлов, не пользовательская история), без падения.
- `meta(key,value)` — `embed_model`, `embed_dim`, `metric`.
- `documents(id, filename, ext, sha256 UNIQUE, added_at, char_count, chunk_count)`.
- `chunks(id, doc_id→documents ON DELETE CASCADE, chunk_index, text, page NULL)`.
- `vec_chunks USING vec0(embedding float[DIM] distance_metric=cosine)`, rowid = `chunks.id`.

### Эмбеддинги (B2)
- Rust-команда `/api/embed` батчем (массив фрагментов за один запрос — не по одному).
- Размерность из ответа; запись в `meta`.
- Отсутствие `bge-m3` → понятное русское сообщение, индексация не выполняется, без паники.

### Чанкинг и индексация (B3)
- Резка по естественным границам: абзацы (`\n\n`) → предложения → набор до целевого размера
  с перекрытием. Числа в константах.
- Идемпотентность: sha256 содержимого файла. Тот же хэш → «уже добавлен». То же имя, другой
  хэш → предложить переиндексацию.
- Прогресс индексации в UI через `Channel` (как стриминг чата), без зависания.

### Поиск (B4) и встраивание (B5)
- `search_documents(query, k=6) → Vec<RetrievedChunk{ text, filename, chunk_index, page, distance }>`.
- Оркестровка во фронте: непустая база → `search_documents` → пакую фрагменты в `CONTEXT`
  (роль `system`, «опирайся только на фрагменты; нет ответа — скажи честно») →
  `[SYSTEM, ...history(урезанная при RAG), CONTEXT, вопрос]` → **неизменённый** `chat_stream`.
- **Бюджет** `num_ctx:8192`: фрагменты добавляются по символьному прокси, пока влезает, с
  резервом под ответ; глубина истории ограничивается при активном RAG. Предотвращает обрыв
  ответа при переполнении контекста.
- **Источники** под ответом известны из шага ретривала до стриминга; сохраняются в сообщение
  ассистента (остаются после перезагрузки диалога).
- **UI базы** — вкладки в сайдбаре «Диалоги | Документы» (по решению заказчика), отдельно от
  списка диалогов; список документов + «Добавить»/«Удалить»; обе темы по CSS-токенам.

### Края (B6)
Пустая база → обычный чат; нет `bge-m3` → сообщение; ничего не найдено → «в документах ответа
нет»; долгая индексация → прогресс; битый файл → честная ошибка.

## 7. Риски и открытые вопросы

1. **`bge-m3` не в провижининге моделей.** Сейчас её установку делает пользователь/разработчик
   вручную. Автоустановка — отдельный этап (как и отмечено в задании). До тех пор путь «модель
   отсутствует» обязателен и проверяется.
2. **Сборка C под Windows (MSVC) и Linux** проверяется только в CI по тегу — у Tauri нет
   кросс-компиляции, локально только macOS. `bundled` SQLite и `sqlite-vec` — это C; особых
   флагов на macOS не понадобилось, для MSVC отметить при первом CI-прогоне.
3. **`rusqlite::Connection` не `Sync`.** Принято открывать соединение на операцию (схема
   создаётся идемпотентно, `CREATE IF NOT EXISTS`) — простая и безопасная модель без глобального
   мьютекса в async-командах.
4. **Время индексации больших документов.** Снижается батч-эмбеддингом и транзакцией; UI не
   виснет за счёт прогресса через `Channel`.
5. **Переполнение контекста («проблема Исаака»).** Контроль — символьный бюджет-пакер + урезание
   истории при активном RAG; нельзя сваливать целые документы.
6. **Качество чанкинга под русский** — ориентировочные числа; тонкая настройка вынесена в
   последующие задания (как и в задании).
